# StreamForge benchmark results

**Headline (this machine):** the engine sustains **~40,000 events/sec end-to-end
at p99 ≤ 25 ms** with 3 workers and exactly-once checkpointing on, and saturates
around **~70,000 events/sec** (p99 climbs to ~450 ms as a Kafka backlog forms).
Raw backlog-drain throughput peaks at **~109,000 events/sec**. Throughput scales
**2.3×** from 1→2 workers but not beyond, because the test VM has only **2 CPU
cores** (`colima start --cpu 2`) — past 2 busy workers the bottleneck is CPU and
shuffle, not partitioning. These are honest single-laptop numbers, not a tuned
cluster.

Single machine (Apple Silicon, Colima/Docker, **2 vCPU / 4 GB**), Redpanda + MinIO, 6 Kafka partitions, 64 key-buckets, 1000ms windows, 2000ms checkpoints. N=300000 events, 500 keys. Engine processing throughput = N / time to consume all N (events/sec); latency = event_time -> aggregation.

## A. Throughput & horizontal scaling (drain a pre-loaded backlog)

Raw processing throughput: a full backlog is pre-loaded, then the engine drains it as fast as it can. (Latency is omitted here — a pre-loaded backlog makes every event 'old' by construction; see experiment B for live latency.)

## B. Latency vs load (3 workers, live injection)

| injected (events/s) | sustained (events/s) | kept up? | p50 latency (ms) | p99 latency (ms) |
|--:|--:|:--:|--:|--:|
| 5000 | 4944 | yes | 2 | 13 |
| 10000 | 9974 | yes | 2 | 6 |
| 20000 | 19922 | yes | 2 | 13 |
| 40000 | 38535 | yes | 4 | 24 |
| 80000 | 71907 | no (backlog) | 24 | 450 |

**Reading it:** the engine keeps up (sustained ≈ injected, low p99) up to ~40k/s;
at 80k/s it can't drain as fast as events arrive, so a backlog builds and tail
latency jumps an order of magnitude — the classic saturation knee. The saturation
point (~70k/s drain) matches experiment A's 2-worker ceiling, i.e. the bottleneck
is per-event CPU work (BadgerDB state + shuffle), addressed by more cores/machines.
